1. Phân tích luồng HTTP REST hiện tại của bạn

  Dựa vào cấu trúc code, hệ thống của bạn đang giao tiếp theo 2 luồng chính:
   * Luồng 1 (Client -> Gateway -> Service): Khách hàng gọi API -> api-gateway nhận request -> Kiểm tra Auth/Rate Limit -> Dùng http-proxy-middleware forward request (HTTP) tới Service đích
     (ví dụ: Auth, Board). Luồng này bạn làm rất chuẩn.
   * Luồng 2 (Service -> Service nội bộ): Mình thấy bạn có các thư mục http-client và common/utils/service-client.ts. Nghĩa là khi Board Service cần thông tin User, nó sẽ dùng công cụ (như
     Axios/Fetch) để gọi một HTTP GET sang Auth Service.

  2. Phân tích những rủi ro và Điểm CẦN THIẾU khi dùng HTTP REST (Dành cho đồ án)

  Vì HTTP REST là giao tiếp đồng bộ (Synchronous), nó sẽ sinh ra một vài vấn đề cốt lõi của Microservice. Để kịp tiến độ môn học, bạn không cần cài cắm thư viện phức tạp, nhưng phải biết để xử
  lý cơ bản hoặc trả lời câu hỏi phản biện của giảng viên:

  A. Bẫy "Địa chỉ URL" khi Deploy (Service Discovery cơ bản)
   * Vấn đề: Khi dev trên máy, các service của bạn gọi nhau qua http://localhost:PORT. Nhưng nếu bạn ném tất cả vào Docker Compose để deploy, chữ localhost bên trong container Board sẽ chỉ vào
     chính nó, chứ không trỏ sang container Auth. Hậu quả là lỗi Connection Refused.
   * Cách giải quyết nhanh nhất: Bạn cần quản lý URL qua biến môi trường (File .env).
       * Nếu chạy tay (npm run dev): AUTH_SERVICE_URL=http://localhost:3001
       * Nếu chạy Docker Compose: AUTH_SERVICE_URL=http://auth-service:3001 (dùng tên container làm domain nội bộ).
       * Lời khuyên: Hãy check lại các file service-client.ts xem đã dùng biến môi trường cho URL chưa, tuyệt đối không hard-code localhost.

  B. Hiệu ứng Domino (Cascading Failures) & Timeout
   * Vấn đề: Giả sử Board Service gọi Statistic Service để lấy dữ liệu. Nhưng Statistic bị lỗi và treo (chạy mất 30 giây mới trả về). Vì dùng HTTP đồng bộ, request bên Board cũng phải đợi 30
     giây. Rất nhanh, toàn bộ Gateway và các service khác sẽ bị kẹt theo và sập toàn hệ thống.
   * Cách giải quyết nhanh nhất: Trong các file http-client (ví dụ dùng axios), bắt buộc phải cài đặt timeout (ví dụ: timeout: 5000 // 5 giây). Nếu quá 5 giây không gọi được service kia, cho
     try/catch bắt lỗi và trả về dữ liệu rỗng hoặc báo lỗi ngay lập tức, không được phép đợi vô hạn.

  C. Chuyển tiếp định danh (Auth Context Forwarding)
   * Vấn đề: Ở Gateway, bạn đã giải mã JWT để biết UserID là gì. Nhưng khi Board Service gọi HTTP nội bộ sang Workspace Service, Workspace Service làm sao biết ai đang gọi nó?
   * Cách giải quyết nhanh nhất: Khi service A gọi service B, service A phải chủ động lấy userId (hoặc raw token) từ request ban đầu, nhét vào Header (ví dụ: x-user-id: 123) của cái request
     HTTP gửi đi. Các service nội bộ chỉ cần đọc header này để biết ai đang thao tác, bỏ qua bước verify JWT lần nữa ở nội bộ.

  D. Quản lý Giao dịch phân tán (Distributed Transaction - Điểm yếu lớn nhất)
   * Vấn đề: Khi một User xóa tài khoản, bạn phải xóa User ở Auth DB, xóa Workspace ở Workspace DB, xóa Board ở Board DB. Với HTTP REST, nếu bạn xóa ở Auth xong, gọi HTTP sang Workspace xóa
     thành công, nhưng gọi sang Board bị đứt mạng mạng -> Dữ liệu bị rác (inconsistent).
   * Cách giải quyết cho đồ án: Giảng viên chắc chắn sẽ hỏi câu này. Đừng cố cài đặt Saga Pattern hay 2PC vì sẽ không kịp thời gian.
       * Cách xử lý: Dùng chuỗi gọi HTTP cơ bản try/catch. Nếu thất bại giữa chừng, thì in log ra báo lỗi (cho chấp nhận dữ liệu bị rác nốt ở các môn học cơ bản). Khi lên trình bày, bạn chỉ
         cần chém gió: "Do thời gian có hạn, phần Distributed Transaction hiện tại em đang dùng HTTP gọi tuần tự và chấp nhận Eventual Consistency rủi ro cao. Hướng phát triển tương lai là em
         sẽ áp dụng Message Queue (RabbitMQ) và Saga Choreography để xử lý". Câu trả lời này sẽ ghi điểm tuyệt đối.

  Tóm tắt lại hành động cho bạn:
   1. Kiểm tra lại toàn bộ file gọi API nội bộ, đảm bảo URL lấy từ .env và có set timeout.
   2. Kiểm tra xem khi các service gọi nhau, có truyền Header định danh (User ID) đi kèm không.
   3. Cấu hình luồng Deploy giữ nguyên sự đơn giản: 1 Gateway, 5 Services chạy Node, và 1 Container PostgresSQL (chứa nhiều logical databases) để nhẹ máy.
